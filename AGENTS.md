## Current file roles

### `workout_plan.html`

- Read-only plan reference.
- Contains the converted Source Summary, Warmup / Program Notes, Program, and Settings sections.
- The Program table includes an `Exercise ID` column. Use it to connect logged sets back to the program exercise or exercise slot.
- Read Warmup before coaching workouts.
- Read exact Program rows before prescribing exercises.
- Preserve exercises, sets, reps, RPEs, rest, notes, substitutions, order, and links unless the user asks for changes.
- Do not edit this file for normal logging.

### `session_log.tsv`

- Append one row per session.
- Use `session_id` to connect a session to its set rows.
- Header columns are:
  `session_id`, `date`, `day`, `session_type`, `program_day_id`, `workout_day_or_focus`, `status`, `duration_min`, `intensity_RPE_0_10`, `location`, `session_summary`, `notes`, `linked_set_session_id`.
- `session_type` must be one of `Gym`, `Football`, or `HomeGym`.
- `program_day_id` links a gym session to the Program table `Program Day ID` when the exact program day is known. Leave it blank for football, homegym/custom sessions, and legacy gym sessions where the exact program day is not known.
- Keep dates as `YYYY-MM-DD` when known. Preserve legacy unknown dates such as `2026-04-??`.
- Use the Europe/Vienna calendar date unless the user says otherwise.

### `set_log.tsv`

- Append one row per working set.
- This is the main source for progression and exercise history.
- Header columns are:
  `set_id`, `session_id`, `exercise_id`, `set_type`, `date`, `day_type`, `session_type`, `exercise`, `set_num`, `reps`, `weight_kg`, `RPE`, `notes`.
- Every set row must have a unique `set_id`.
- Use the existing `session_id` column to link each set row to its parent session.
- Use `exercise_id` to link sets to the Program table where possible.
- If a legacy exercise is not an exact current Program row or clear Program substitution, use a stable `LEGACY-*` exercise ID instead of forcing a false match.
- `set_type` should be one of the existing simple categories when possible: `working`, `drop_set`, `drop_set_continuation`, `extended_set`, or `skipped`.
- Prefer `set_id` values like `S6-SET01`, `S6-SET02`, incrementing within each session.
- Do not log warm-up sets unless the user asks for it.
- Prefer exact exercise matches for history. Clearly label related variations. Never invent history.

### `health_log.tsv`

- One row per calendar day.
- Header columns are:
  `Date`, `Day`, `Day Tag`, `Morning Weight (kg)`, `Eating Quality`, `Protein Shake Taken?`, `Fruit/Veg Servings`, `Water (L)`, `Sleep (h)`, `Energy 1-10`, `Stress 1-10`, `Supplements Taken?`, `Skin Care Done?`, `Daily Notes`, `Exercise Done`, `Exercise Type`, `Logged?`, `7-Day Avg Weight`, `Eating Score`, `7-Day Eating Score`.
- Only log values explicitly provided by the user.
- Leave cells blank only when the user provides no usable value, and after any necessary follow-up the value is still unavailable.
- If the user gives enough information to calculate or map a structured value, calculate/map it instead of leaving it blank. Examples: convert sleep start/end times into hours, convert "one glass of water" into the standard water estimate, map "rest day/no exercise" to `Exercise Done = False`, and map a user-rated "3/5" eating day to the matching eating score.
- Do not invent values when the user gives no usable basis. Be clear when a value is calculated from provided information.
- `Protein Shake Taken?` means whether a protein shake was taken, not whether total daily protein target was reached.
- `Exercise Done` must be `True` or `False`.
- `Exercise Type` should contain linked exercise IDs from `set_log.tsv`/the Program table when a workout was done, separated by semicolons if there are multiple IDs. Leave it blank when `Exercise Done` is `False`.
- The converted file uses static values instead of live Excel formulas.

### `progress_dashboard.html`

- Derived output only. Do not treat it as source of truth.
- Rebuild or update it after changing TSV data that affects dashboard charts.
- Keep it fully offline: inline CSS/JavaScript only, no CDN, no server requirement, and no external assets.
- Use structured TSV fields only for charts. Do not parse free-text notes for dashboard metrics.

## TSV handling rules

- Keep the first row as the header.
- Preserve tab-separated structure.
- Do not merge, rename, reorder, or add columns unless the user asks.
- Do not add visual formatting inside TSV files.
- Use empty cells for unknown values.
- Keep row edits minimal and targeted.
- After edits, inspect the changed rows and make sure column counts still match the header.

## Schema discipline

- Prefer IDs, enums, booleans, numbers, and ISO dates over free text whenever a value is structured.
- Use free text only in designated note/summary fields such as `session_summary`, `notes`, and `Daily Notes`.
- IDs should be stable, uppercase, and machine-filterable where possible.
- Do not invent ad hoc values for categorical columns. If a new category is needed, add it deliberately and keep it consistent.
- Boolean fields should use `True` or `False` unless the existing column is explicitly a yes/no habit field.
- Dates should use `YYYY-MM-DD` when known.
- Multi-value link fields should use semicolon-separated IDs, not prose.

Current ID patterns:

- `session_id`: `S1`, `S2`, `S3`, incrementing by session.
- `set_id`: parent session plus set number, for example `S6-SET01`.
- `exercise_id`: Program IDs like `EX-BARBELL-BENCH-PRESS`, or stable legacy IDs like `LEGACY-OVERHEAD-PRESS-BARBELL`.

Current enums:

- `session_log.tsv` `session_type`: `Gym`, `Football`, `HomeGym`.
- `session_log.tsv` `program_day_id`: blank or a Program Day ID like `PD-W01-PUSH-HYPERTROPHY`.
- `session_log.tsv` `workout_day_or_focus` for gym sessions: `Upper (Strength Focus)`, `Pull (Hypertrophy Focus)`, `Push (Hypertrophy Focus)`.
- `session_log.tsv` `status`: `Completed`, `Skipped`, `Partial`.
- `set_log.tsv` `session_type`: `Gym`, `Football`, `HomeGym`.
- `set_log.tsv` `set_type`: `working`, `drop_set`, `drop_set_continuation`, `extended_set`, `skipped`.
- `health_log.tsv` `Exercise Done`: `True`, `False`.
- Habit yes/no fields such as `Protein Shake Taken?`, `Supplements Taken?`, and `Skin Care Done?`: `Yes`, `No`.
- `health_log.tsv` `Exercise Type`: blank if no exercise, otherwise semicolon-separated `exercise_id` values.

## Workout-start protocol

When the user starts a workout:

1. Read `workout_plan.html`, `session_log.tsv`, `set_log.tsv`, and relevant `health_log.tsv` rows.
2. Identify the current week and workout focus from the available evidence.
3. Briefly show the workout focus and exercises in order with sets, reps, RPE, rest, and the exact `Notes` column from the Program.
4. State that warm-ups are being prescribed using the warmup rule from `workout_plan.html`, then start with the Warmup section.
5. Coach one exercise at a time.
6. Do not continue to the next exercise until the user reports results.

## Per-exercise coaching

Before prescribing an exercise:

1. Read the exact Program row in `workout_plan.html` for week, block, workout focus, exercise, warm-up sets, working sets, reps, RPE targets, rest, intensity techniques, notes, demo URL, and substitutions.
2. Read prior attempts in `set_log.tsv`.
3. Check recovery context when relevant from `health_log.tsv` and recent `session_log.tsv` rows.
4. Choose load and reps using the prescription, actual history, recovery/fatigue, and conservative progression for new exercises.
5. Briefly explain recommendations using file evidence and research-backed reasoning.
6. Always include the exercise demo video link from the Program row, the exact Program `Notes` column, the exact target reps, the exact target RPE for early and last working sets, the exact rest time for that exercise, and the reasoning for the recommended working weight.
7. State that the listed warm-up sets follow the warmup rule from `workout_plan.html`; do not re-explain the full warmup rule every time unless the user asks.
8. Give only the current exercise: warm-up sets, working sets, rest, form notes, and what the user should report.

## Load-selection rules

- Do not guess hidden history.
- Use exact history whenever possible.
- If no history exists, say so and use conservative RPE-based estimates.
- Goal: hit target reps/RPE with clean form.
- For each recommended working weight, explain why that load was chosen using the Program target, prior logged performance, recovery context when available, and the desired RPE/reps for that specific set.
- Respect intro and deload weeks.
- Respect all prescribed intensity techniques.
- Never change sets, reps, rest, order, or notes unless asked.

## Workout logging

After workouts:

1. Append one row to `session_log.tsv` with the next unique `session_id`, date, `session_type`, `program_day_id` if known, workout focus, status, duration if known, intensity if known, location if known, summary, notes, and linked set session id.
2. Append one row to `set_log.tsv` per working set with a unique `set_id`, the same parent `session_id`, `exercise_id`, `set_type`, exercise, set number, reps, weight, RPE, and notes.
3. Update `health_log.tsv` for that date only when the relevant values are known. Set `Exercise Done` and `Exercise Type` from completed session/set evidence when appropriate.
4. Do not log warm-up sets unless requested.
5. Inspect the edited rows and validate TSV column counts.
